# wxyc-canary

Synthetic-DJ canary on AWS Lambda. Probes the WXYC user-facing API every five minutes and emits CloudWatch metrics + alarms. Built after the 2026-04-30 triple-incident (catalog-search 503, flowsheet POST 500, iOS decoder drift) to close the gap where production failures were detected by users rather than monitors.

## Conventions

- Each check has a stable kebab-case `name` that lands in CloudWatch as a `Check` dimension. Renaming breaks pinned dashboards — pick well first time.
- `CheckFailure` is published twice per check: once with the `Check` dimension (for dashboards and per-surface drill-down) and once dimensionless (a cross-check rollup for dashboards). The dimensionless `CheckFailure` _used_ to be the series the `wxyc-canary-check-failure` alarm queried; as of wxyc-canary#48 the page reads the narrower `UserFacingCheckFailure` aggregate instead (see the paging-tier bullet below). Emit-twice is still the rule for any metric an alarm reads: CloudWatch alarms reject the `SUM(SEARCH(...))` expression that would otherwise let one alarm watch every dimension value (commits b7312c3 / 299deb5 / issue #13), so a dimensionless companion is how a plain-form alarm reads a metric we also publish dimensioned. The alarms use the plain `Namespace`/`MetricName` form with `Statistic: Maximum` so any failing check in a 5-minute window trips that check's tier alarm (a user-facing check trips `wxyc-canary-check-failure`; an infra check trips `wxyc-canary-infra-degraded`). `CheckSkipped` and `CheckLatency` are emitted dimensioned-only — they're for dashboards, not alarming. The `template.yaml ↔ publishMetrics contract` test in `test/handler.test.ts` parses `template.yaml`, harvests the metrics the handler actually publishes, and asserts every `Namespace: WXYC/Canary` alarm targets a `(MetricName, Dimensions-shape)` the code emits — adding a new canary alarm therefore requires either matching the emission or adding one. External-namespace alarms (`AWS/Lambda`, `WXYC/BackendService`) are skipped.
- **Paging tier (`pagesOncall`, wxyc-canary#48).** Each check's failure value is _also_ routed into exactly one dimensionless aggregate: `UserFacingCheckFailure` — the series the `wxyc-canary-check-failure` page reads (action → `AlertTopic`) — or `InfraCheckFailure`, the low-urgency series the `wxyc-canary-infra-degraded` alarm reads (action → `InfraAlertTopic`, which is **console-only until `InfraAlertEmail` is set**). The split is driven by an explicit `pagesOncall` field on the check definition (default `true` — fail-safe, a new check pages until someone opts it out), and is **deliberately NOT derived from `suites`**: `dj-rotation` / `dj-rotation-picker` are untagged (CLI-unreachable) yet user-facing and must page, while the two infra/CI probes `gha-runner-online` and `semantic-index-freshness` carry `pagesOncall: false`. (`semantic-index-search` was demoted to infra for the nightly-blip window but is back on the page now that semantic-index#347 ended the in-process OOM-restart — wxyc-canary#50. `semantic-index-freshness` stays non-paging: a stale graph is degradation, not an outage; promoting it is a separate decision, not gated on #50.) `suites` is a CLI-reachability axis only; paging tier is `pagesOncall`. Both aggregates are dimensionless-only on purpose — per-surface drill-down is already served by the dimensioned `CheckFailure`, so a `Tier` dimension or a second dimensioned emission would add cost for no benefit. The classification-pin test in `test/checks.test.ts` guards the tier assignment; the `publishMetrics — tier split` block in `test/handler.test.ts` pins the routing.
- Custom per-check metrics (e.g. `EnrichmentLagSeconds` from `enrichment-quality`, `GraphDbAgeSeconds` from `semantic-index-freshness`) follow the same emit-twice pattern. A check declares them by returning `{ metrics: { ... } }` from its `run`; the runner stores them on the outcome; `publishMetrics` emits each entry once dimensioned (`Check=<name>`) and once dimensionless. Unit is inferred from the metric-name suffix: `*Seconds` → `Seconds`, `*Milliseconds` → `Milliseconds`, else `Count`. Keep names consistent with this convention — a metric named `Latency` would silently publish as `Count` and break any unit-aware dashboard. Not every custom metric needs an alarm: `GraphDbAgeSeconds` is dashboard-trend-only (the freshness signal an operator acts on is the `InfraCheckFailure` aggregate), so the `template.yaml ↔ publishMetrics contract` test (one-directional: alarms must target an emitted metric, but the code may emit alarm-free metrics) stays green without a dedicated age alarm.
- Checks fail by **throwing**. The runner converts the throw into a `CheckOutcome` with `status: 'fail'` and the message. One throw never short-circuits the others (parallel `Promise.all` of independent try-catches).
- `requiresAuth: true` checks downgrade to `skipped` (separate metric from `fail`) when no DJ credentials are configured. The alarm only fires on `fail`, so an operator who hasn't provisioned the DJ test account gets a noisy console but not a phone call.
- `writes: true` checks (the v1 write canary) downgrade to `skipped` when `CANARY_ENABLE_WRITE_PROBE=true` is unset. Gate is intentional: a fresh deploy stays read-only until the operator provisions the DJ test account and reviews cleanup-on-failure behaviour. A check can also return `{ skipped: true, skipReason }` to record skip from inside its `run` (used by the enrichment check when another DJ is on-air — the canary deliberately does not insert sentinel rows into a live show).
- Two credential paths: `CANARY_DJ_EMAIL` + `CANARY_DJ_PASSWORD` env vars (local + tests) or `CANARY_DJ_SECRET_ARN` pointing at a Secrets Manager secret with `{"email":"...","password":"..."}` (prod).
- Retry policy: the canary does **not** retry the surfaces it measures. The lone carve-out is `signInDj` retrying once on 429 only — auth is a precondition shared by 7 of the 12 checks, so a single 429 there cascades into 7 simultaneous fail outcomes plus a Lambda Errors alarm. The retry honors `Retry-After` (seconds form), capped at 5s to fit the Lambda budget. Token exchange is not retried.
- OIDC probe (`oidc-authorize`, wxyc-canary#60 + #63): the check reads the session token via `ctx.djAuth.sessionToken` (the `signed-in` narrowing of the `DjAuthState` discriminated union — wxyc-canary#65) — the raw better-auth session token from `/sign-in/email` — and passes it as `Authorization: Bearer ...` on `/oauth2/authorize`, because the JWT from `/token` is a different audience and the authorize endpoint rejects it. The bearer plugin (`node_modules/better-auth/dist/plugins/bearer/index.mjs`) translates the header into the session cookie the authorize path reads via `getSessionFromCtx`. Probes are folded into `ctx.oidcProbes` — a non-empty `readonly [OidcProbe, ...OidcProbe[]]` tuple of `(clientId, redirectUri, label)` triples (wxyc-canary#63). The env loader today produces exactly one probe (the `wxyc-canary` PUBLIC trusted client registered in WXYC/Backend-Service#1576); the tuple shape lets a second consumer (WikiJS, other in-house tools) register by config alone. The check iterates every probe on each tick, aggregates failures with `[label]` prefixes, and does NOT short-circuit — one tick catches multi-client damage in one alarm. Public client + PKCE means no `client_secret` lives in the canary env — the probe stops at the 302, never exchanges the code, and a leaked canary env carries no OIDC credential. `redirect: 'manual'` on `canaryFetch` gets Node fetch to return the 302 Location without following it (see `src/client.ts`; the `location` accessor is the typed reader for that Location — wxyc-canary#64). The check publishes no custom metrics — its cost signal is the standard `CheckLatency`. Extracted to `src/oidc-authorize.ts` in wxyc-canary#62; details live there.
- The GitHub-issue reporter is **best-effort and non-fatal**. The handler wraps `reportOutcomesToGitHub` in a try/catch (parallel to `publishMetrics`) — a GitHub outage logs and continues so the canary's primary signal (the tier aggregates `UserFacingCheckFailure` / `InfraCheckFailure` + the Lambda Errors alarm) is never masked by a reporter-induced exception. The SNS topic and `wxyc-canary-lambda-errors` alarm stay armed as the safety net for when the Lambda dies before the reporter runs. Dedup is by label (`canary:check:{name}`), not title — titles drift as error messages change; labels are durable. A `pass` after a `fail` closes the open issue with a recovery comment; a `pass` with no open issue is a no-op. `skipped` outcomes are always no-ops (operator gap, not regression).
